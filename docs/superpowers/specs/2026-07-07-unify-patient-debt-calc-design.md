# Дизайн: единый источник правды для долга/баланса пациента и оплаты приёма

**Дата:** 2026-07-07
**Контекст:** пункт «Шаг 2.3» из аудита архитектуры (`akmsoft_audit`, 2026-07-06) — «тройная параллельная логика расчёта долга». Коммит `f23d95a` уже схлопнул view-уровень (`finance/views.py:_recalc_patient_balance`) в вызов `patient.recalc_balance()`. Этот документ закрывает оставшуюся часть пункта.

## Проблема

Сейчас параллельно существуют независимые реализации:

1. `Patient.debt` (свойство, `apps/patients/models.py:239-253`) — считает долг живым запросом при каждом обращении: `max(0, (total-discount) - (income-refund))`, по треим `self.treatments` (менеджер, фильтрующий по текущей клинике).
2. `Patient.balance` (поле) + `Patient.recalc_balance()` (`apps/patients/models.py:255-274`) — та же формула (с обратным знаком), но пишет в денормализованное поле `balance`. Явно игнорирует фильтр клиники (`Treatment.all_objects...`, докстринг «по всем клиникам»). Вызывается почти из всех точек мутации: `Payment.save()`/`Payment.delete()` (`finance/models.py`), `treatment_set_discount`, `treatment_status`, `Treatment.recalculate_total()`.
3. `Payment._update_treatment_paid()` (`finance/models.py:101-107`) — считает `paid_amount` приёма напрямую суммой `Payment` на этот приём, **игнорируя `PaymentAllocation`**.
4. `_recompute_treatment_paid()` (`finance/views.py:100-105`) — считает то же самое, но через `PaymentAllocation` — может разойтись с (3), если один платёж покрывает несколько приёмов.
5. `PaymentSerializer.create()` (`finance/serializers.py:21-23`) — создаёт `Payment` без вызова `_allocate_income()`. Модалка распределения платежа (`payment_allocations`) для API-платежей всегда пустая.

## Ключевая находка

Формулы (1) и (2) математически идентичны: `debt == max(0, -balance)`. `balance` уже надёжно поддерживается свежим во всех точках мутации данных, которые на него влияют (создание/правка/удаление платежа, изменение скидки приёма, смена статуса приёма, пересчёт суммы приёма). Значит `debt` можно свести к чтению `balance` без единого запроса, не меняя формулу.

`_allocate_income()` (в `finance/views.py`) уже идемпотентна (удаляет и пересоздаёт распределения для платежа), не зависит от `request`/view-контекста — только от `payment` и данных БД. Её можно безопасно вызывать из `Payment.save()`, что автоматически чинит API-путь без правок в сериализаторе.

## План изменений (сужено 2026-07-07 после повторного анализа рисков)

Изначальный план включал также перевод `Treatment.paid_amount` на allocation-based расчёт (пункты 2-3 ниже вычеркнуты). При повторном анализе выяснилось: allocation-based расчёт (`_recompute_treatment_paid`, views.py) **не учитывает возвраты** (`PaymentAllocation` создаётся только для `type=income`, для `refund` аллокаций не существует), тогда как текущий прямой расчёт (`Payment._update_treatment_paid()`) возвраты учитывает корректно, но не видит расщепление платежа на несколько приёмов. Перевод на allocation-based без отдельного механизма аллокации возвратов — тихая регрессия учёта возвратов на живых данных. Отложено как отдельный, более рискованный пункт (см. «Не в рамках»).

Также обнаружена ordering-зависимость в `payment_create`: скидка на приём применяется **после** первого `payment.save()`, а существующий повторный явный вызов `_allocate_income()` в конце view пересчитывает распределение уже с учётом скидки. Явные вызовы в `views.py` поэтому **не удаляются** — они идемпотентны, повторный вызов безвреден, а их удаление рисковало бы пересчитать распределение по устаревшей сумме приёма.

Финальный узкий план:

1. **`apps/finance/models.py`** — перенести `_allocate_income(payment)` и `_recompute_treatment_paid(treatment)` из `views.py` в `models.py` как модульные функции (та же логика, без изменений). В `Payment.save()`: после существующих `_update_patient_balance()`/`_update_treatment_paid()` вызывать `_allocate_income(self)`, если `self.type == Payment.TYPE_INCOME`. Это единственное новое поведение — до сих пор эта функция вызывалась только явно из `views.py` (веб-путь), теперь она вызывается для любого сохранения income-платежа, включая DRF API.
2. **`apps/finance/views.py`** — удалить локальные определения `_allocate_income`/`_recompute_treatment_paid`, импортировать их из `models.py`. Явные вызовы `_allocate_income()`/`_recalc_patient_balance()` в `payment_create`/`payment_edit`/`payment_delete` — **оставить без изменений** (идемпотентны, нужны для корректного порядка со скидкой).
3. **`apps/patients/models.py`** — заменить тело свойства `Patient.debt` на `return max(Decimal(0), -self.balance)`.

`Payment.delete()` и `Payment._update_treatment_paid()` (прямой расчёт `paid_amount` приёма) — **не трогаются вообще** в этом раунде.

## Единственное фактическое изменение поведения

Пациент и приём — каждый принадлежит ровно одной клинике (`clinic` FK, проставляется автоматически той клиникой, что была активна при создании записи). Суммы никогда не смешиваются между разными клиниками для одного пациента — у пациента физически нет приёмов «от другой клиники».

Разница между `self.treatments` (менеджер `objects`, фильтрует по клинике, активной в сессии смотрящего) и `Treatment.all_objects.filter(...)` (без этого фильтра) проявляется только в редком кросс-клиничном случае: суперадмин переключён на клинику A, но открывает карточку пациента, принадлежащего клинике B. Старое свойство `debt` (через `objects`) в этом случае **некорректно** отфильтровало бы приёмы пациента B по активной клинике A и посчитало бы долг неверно. `recalc_balance()` (через `all_objects`) считает правильно — берёт приёмы именно этого пациента независимо от того, что активно у смотрящего.

Вывод: унификация — это **исправление бага** старого `debt` для кросс-клиничного случая, а не смена бизнес-формулы. Для обычного сценария (пациент и смотрящий в одной активной клинике) числа не меняются вообще. Подтверждено пользователем 2026-07-07.

## Риски и проверка

- Формулы (1)=(2) доказаны математически эквивалентными — риск регресса в расчёте суммы низкий.
- `paid_amount` приёма и `Payment.delete()` не меняются — риск регрессии учёта возвратов снят исключением этого пункта из объёма (см. «Не в рамках»).
- Единственное новое поведение — `Payment.save()` теперь всегда вызывает `_allocate_income()` для income-платежей (раньше — только там, где view явно её вызывал). Для веб-пути это не меняет ничего (там уже был явный вызов). Для API-пути это добавляет создание `PaymentAllocation`, которых раньше не было вообще — по определению не может «сломать» то, чего не было.
- Ручная проверка после изменений: создать платёж через веб-форму и через DRF API, в обоих случаях открыть модалку распределения платежа (`payment_allocations`) и убедиться, что она не пустая в обоих случаях; изменить скидку на приём через `payment_create` и убедиться, что распределение считается по сумме после скидки (не до); проверить, что `Patient.debt` и `Patient.balance` дают одинаковое число на нескольких реальных пациентах локально перед выкладкой.
- Тестов на денежную логику в проекте нет вообще (отмечено в аудите). План реализации добавляет unit-тесты Django `TestCase` на все изменённые точки — это первые тесты в проекте.

## Не в рамках этого дизайна

- Обёртывание денежных операций в `transaction.atomic()` (Шаг 2.2 аудита) — отдельный пункт.
- `clinic` в моделях `warehouse`/`medicines` (Шаг 2.1 аудита) — отдельный пункт.
- Перевод `Treatment.paid_amount` на allocation-based расчёт — требует сначала отдельного механизма аллокации возвратов (`allocate_refund`), иначе тихо ломает учёт возвратов. Отложено до отдельного раунда с более тщательным дизайном и тестами на возвраты.
- `Payment.delete()` — не трогается, пока не трогается и `paid_amount`.

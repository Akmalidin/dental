## 1. 🔴 План лечения: неверный «Итого по плану» (количество и скидка игнорируются)

**Файл:** `apps/treatments/models_plan.py:41-45`

```python
@property
def total_price(self):
    return self.items.aggregate(s=Sum("price"))["s"] or Decimal(0)   # ← только цена за 1 шт., без qty и discount
```

А строка и этап считаются правильно:

```python
# TreatmentPlanItem.subtotal (models_plan.py:132)
gross = self.price * self.quantity
return gross - (gross * self.discount / Decimal(100))   # с количеством и скидкой
```

**Последствие:** в строках таблицы и в «Итого по этапу» суммы с учётом количества/скидки, а нижнее **«Итого по плану» — без них**. Как только у пункта `Кол-во > 1` или есть скидка, итог занижается и не сходится со строками.

**Где видно пользователю:**
- Страница пациента → вкладка «План лечения», футер таблицы — `templates/patients/detail.html:467` (`plan.total_price`).
- Редактор плана — `templates/treatments/plan_detail.html:89` (`plan.total_price`), при том что строки выше используют `item.subtotal` (стр. 64) и этап — `stage.total` (стр. 42). **На одной странице две несходящиеся цифры.**
- Печать плана — `apps/treatments/views.py:520`.
- Тем же багом страдает `done_price` (стр. 47-51) — прогресс по деньгам.

> На проде план #3 содержит 1 услугу с qty=1, скидка 0 — поэтому визуально совпало (20000=20000). Баг проявляется на любом плане с количеством >1 или скидкой.

**Фикс:** считать `total_price`/`done_price` через сумму `subtotal` пунктов (как в `stage.total`), а не `Sum("price")`.

## 3. 🔴 Безопасность: XSS / HTML-инъекция в печати плана и медкарты

**Файлы:** `apps/treatments/views.py` — `plan_print` (502-522) и `treatment_emr_print` (595-622).

HTML собирается f-строками, пользовательские данные вставляются **без экранирования**:

```python
rows += f"<tr><td>{it.service.name}</td>..."        # plan_print
... f"Пациент: <b>{plan.patient.full_name}</b> · Врач: {plan.doctor.name} ..."
body += f"<p><b>{label}:</b> {val}</p>"               # treatment_emr_print: текст жалоб/анамнеза/диагноза
```

Если в названии услуги, имени пациента, заметке или тексте ЭМК есть `<script>…</script>` (или `<img onerror=…>`), он исполнится при открытии печати. Вектор реалистичен: имена/услуги/медкарту вводят операторы и врачи.

**Фикс:** рендерить через Django-шаблон (автоэкранирование) либо оборачивать значения в `django.utils.html.escape()`.

## 4. 🟠 Страница пациента: статусы зубов — недоделанная/мёртвая функция

В системе есть всё для зубной карты со статусами (кариес, пломба, удалён, цвета):
- модель `ToothCondition` + справочник `ToothStatus` (`apps/treatments/models_teeth.py`);
- эндпоинт сохранения `patient_tooth_set` (`apps/patients/views.py:245`), подключён в urls как `/<pk>/tooth-set/`;
- справочник статусов на `/settings/tooth-statuses/` (открывается, 200).

**Но на странице пациента это не работает:** в `templates/patients/detail.html` зубная карта (`x-data` → `teeth: []`) инициализируется **пустой** и существующие `ToothCondition` не загружает. Поиск показал **0 обращений** к `patient_tooth_set`/`ToothCondition` из шаблонов и JS — эндпоинт не вызывается ниоткуда.

**Итог:** карта зубов на странице пациента служит только для выбора зубов под услугу. Поставить/увидеть постоянный статус зуба (кариес/пломба/удалён) из карточки пациента **нельзя** — справочник статусов есть, а применить их негде. Это и есть «непонятная страница»: функция выглядит готовой, но разрознена и не доведена.

**Фикс (выбрать):** либо подключить карту к `ToothCondition` (загрузка цветов + клик → `tooth-set`), либо убрать незавершённый справочник, чтобы не путать.

## 5. 🟠 Баланс пациента может устаревать

`patient.debt` (`apps/patients/models.py:122`) считается на лету — всегда корректен.
`patient.balance` — **хранимое поле**, обновляется только вызовом `recalc_balance()`.

Проверено: `Payment.save()` зовёт `recalc_balance()` (`apps/finance/models.py:77`), создание/редактирование приёма — тоже. **Но `Payment.delete()` пересчёт не вызывает** (в модели только `save`). Удаление платежа оставит «+баланс»/долг в сайдбаре устаревшим (`detail.html:96` использует хранимое `patient.balance`).

**Фикс:** переопределить `Payment.delete()` (и массовые удаления) с вызовом `recalc_balance()`, либо везде в UI показывать вычисляемое значение.

## 6. 🟡 Страница пациента: inline-WhatsApp в карточке сломан

`templates/patients/detail.html:839-852` — `sendNotify()` шлёт POST на `/patients/<pk>/notify/` и ждёт JSON `{ok:true}`. Но view `patient_notify` (`apps/patients/views.py:156`) возвращает **HTML-страницу/redirect, не JSON** → `await r.json()` падает → пользователю всегда «Ошибка сети», даже если сообщение ушло.

Полноценная страница `/patients/<pk>/notify/` работает отдельно (открывается, 200) — ломается именно быстрый inline-вызов (по сути мёртвый код в текущем шаблоне).

**Фикс:** удалить неиспользуемый `sendNotify` или сделать в `patient_notify` JSON-ответ при AJAX-запросе.
